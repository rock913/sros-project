## 初步分析 (Analysis)

### 当前问题

用户报告：
- 点击 "📋 View Details" 按钮后无反应
- `Decision Type: null` 表示后端数据可能不完整
- 模态对话框不够友好，需要更好的交互方式

### 现有实现分析

#### 1. 当前 HITL 处理流程
```typescript
// extension.ts - createHitlRequestHandler()
vscode.window.showWarningMessage(
    `⏸️ HITL Approval Required\n\n${hitlData.prompt}`,
    { modal: true },
    '✅ Approve',
    '❌ Reject',
    '📋 View Details'
).then(selection => {
    if (selection === '📋 View Details') {
        // 创建新的 WebView Panel
        const hitlPanel = vscode.window.createWebviewPanel(...);
    }
});
```

**问题**：
1. ❌ 用户需要在对话框和详情面板之间切换
2. ❌ 详情面板是独立窗口，割裂了体验
3. ❌ 没有利用已存在的 Research Progress 面板

#### 2. 后端 HITL 数据结构

检查后端发送的数据：
```json
{
  "type": "hitl_request",
  "request_id": "hitl_query_approval_xxx",
  "decision_type": null,  // ❌ 可能是 null
  "prompt": "AI已为研究主题「recent advance on vllm」生成以下查询，是否继续？",
  "options": ["approve", "reject", "modify"],
  "context": {
    "research_topic": "recent advance on vllm",
    "queries": [...]
  }
}
```

### 优化方案

**核心思路**：将 HITL 审批 UI 直接嵌入到 Research Progress 面板，而非弹窗。

**优势**：
1. ✅ 单一界面，无需切换
2. ✅ 保持上下文连续性
3. ✅ 可以展示更丰富的信息（查询列表、上下文）
4. ✅ 支持内联操作（Approve/Reject/Edit）

## 调试快照 #1: WebSocket 过早关闭问题

### Error
用户报告测试结果：
```
10:10:49 PM ⏸️ Waiting for query approval...
10:10:49 PM ⏸️ Waiting for your approval...
10:10:49 PM ⏸️ Waiting for human approval...
10:10:49 PM 🔄 agent_complete  ← 立即显示完成！
10:10:49 PM ✅ Research completed!
10:11:03 PM ✅ Approved - Research continuing...  ← 14秒后点击批准
```

**问题**：HITL 触发后立即显示 "Research completed!"，用户无法有效审批。

### Hypothesis

后端日志显示：
```log
[HITL] Sent request hitl_query_approval_7e0ae055 to frontend
INFO:     connection closed  ← WebSocket 立即关闭！
```

**根本原因**：
1. 后端发送 HITL 请求后，WebSocket 连接立即关闭
2. 后端认为研究已完成，发送 `complete` 消息
3. 前端收到 `complete`，显示 "Research completed!"
4. 用户的审批响应无法发送（连接已关闭）

**可能的后端问题**：
- 后端 HITL 配置可能设置了超时=0（立即跳过）
- 后端 HITL 实现有 bug（未正确等待响应）
- 后端环境变量配置跳过了 HITL

### Fix Attempt

**临时方案**（按用户要求）：禁用前端 HITL 交互，自动批准所有请求。

**修改内容**：
```typescript
// vscode-extension/src/extension.ts
function createHitlRequestHandler(...) {
    return (hitlData: any) => {
        // ⚠️ TEMPORARY: Auto-approve all HITL requests
        const { sendHitlResponse } = require('./api');
        sendHitlResponse(wsConnection, hitlData.request_id, true, hitlData.options?.[0]);
        
        // Update progress
        progressPanel.webview.postMessage({
            command: 'updateProgress',
            message: '✅ Auto-approved! Continuing research...',
            progress: 40
        });
    };
}
```

**效果**：
- ✅ 研究流程不会中断
- ✅ 无需用户手动审批
- ✅ HITL 请求自动批准
- ⚠️  失去了人工审核的安全机制

## Debugging Snapshot #2: WebSocket Error Handling Improvement

**Date**: 2025-11-03 22:20
**Issue**: `❌ Error: Failed to parse server message` 误导用户
**Root Cause**: `ws.on('message')` catch 块通用错误处理

### 问题分析

**错误时间线**:
```
10:16:28 PM ⏸️ Waiting for query approval...
10:16:28 PM ❌ Error: Failed to parse server message  ← 误导性错误
10:16:28 PM 🔄 agent_complete
10:16:28 PM ✅ Research completed!
```

**实际发生顺序**:
1. 后端发送 `{"type": "hitl_request", ...}`
2. 前端自动批准（发送 `hitl_response`）
3. **后端立即关闭 WebSocket**（bug）
4. 前端尝试解析已关闭连接的数据
5. JSON.parse() 失败 → catch 块 → 向用户显示错误

**后端日志证据**:
```log
[HITL] Sent request hitl_query_approval_9ccf5394 to frontend
INFO:     connection closed  ← 立即关闭，未等待响应
```

### 修复方案

**文件**: `vscode-extension/src/api.ts`

**修改 1**: 移除解析错误的回调
```typescript
// 修改前：
catch (error: any) {
  console.error('[WebSocket] Parse error:', error);
  callbacks.onError?.('Failed to parse server message'); // ← 向用户显示错误
}

// 修改后：
catch (error: any) {
  console.error('[WebSocket] Parse error:', error);
  console.error('[WebSocket] Raw data:', data.toString());
  // 不要为解析错误调用 onError - 连接可能正在关闭
  // 只记录错误日志
}
```

**修改 2**: 增强关闭事件日志
```typescript
ws.on('close', (code, reason) => {
  const reasonStr = reason?.toString() || 'No reason provided';
  console.log(`[WebSocket] Connection closed: Code ${code} - ${reasonStr}`);
  
  if (code !== 1000 && code !== 1001) {
    console.warn(`[WebSocket] ⚠️ Unexpected closure code: ${code}`);
    // 注意：这通常发生在后端发送 HITL 请求后
    // 但在接收 hitl_response 之前关闭连接。这是后端 bug，不是前端错误。
  }
});
```

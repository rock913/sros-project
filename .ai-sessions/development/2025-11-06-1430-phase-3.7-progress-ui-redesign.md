1. **File Creation Issues**: 
   - `create_file` tool had issues with large template literals
   - Solution: Used Python script via terminal for reliable file creation

2. **Template Literal Escaping**:

   - No hardcoded colors

```
│  ┌────────────────────────────────────────┐         │

│  │  - Data transformation                 │         │
│  │  - Error handling                      │         │
│  └──────────────┬─────────────────────────┘         │
│                 │                                    │
└─────────────────┼────────────────────────────────────┘
                  │ HTTP (fetch)
                  │
┌─────────────────▼────────────────────────────────────┐
│            Backend API (FastAPI)                     │
├─────────────────────────────────────────────────────┤
│  GET  /sessions                                      │
│  GET  /sessions/{id}/details                         │
│  POST /sessions                                      │
│  GET  /sessions/{id}/export?format=md|pdf           │
│  GET  /papers/{id}/bibtex                           │
└─────────────────────────────────────────────────────┘
```

**Message Flow**:
```
User Click → Webview JS → postMessage() 
    ↓
Provider.handleMessage() → Route to handler
    ↓
Handler → Call apiClient function
    ↓
apiClient → HTTP Request to Backend
    ↓
Backend Response → Transform data
    ↓
Provider.updateView() → Re-render HTML
    ↓
Webview displays updated content
```

## 📝 Technical Decisions Made

### 1. **API Compatibility Layer**
Instead of modifying backend API, we transform data in the provider:
```typescript
session_id → id
papers_count → paper_count
active → in_progress
```

**Rationale**: Keeps backend unchanged, maintains backward compatibility

### 2. **State Management Strategy**
State stored in provider class properties:
- `_sessions`: Session[]
- `_selectedSessionId`: string | null
- `_sessionDetails`: SessionDetails | null

**Rationale**: Simple, centralized, no additional state library needed

### 3. **HTML Regeneration on Update**
Every state change triggers full HTML regeneration:
```typescript
this._updateWebview() → webview.html = generateHTML(...)
```

**Rationale**: Simpler than DOM manipulation, webview is isolated

### 4. **Error Handling Pattern**
Try-catch with dual logging:
```typescript
try {
    // API call
} catch (error) {
    console.error('...', error);  // Developer
    vscode.window.showErrorMessage(...);  // User
}
```

**Rationale**: Both developer debugging and user feedback

## Architecture Analysis (Reflection)
**Why Session Files?**
In an AI-Native system (specifically one using stateless LLMs like Copilot), **Session Files act as External Memory (Hippocampus)**. They bridge the context gap between interaction turns and between different agents (Architect -> Builder).
- **Alternative considered**: Storing state in a database or issue tracker.
- **Decision**: Structued Markdown files are currently superior because:
    1. They are "Token-Native" (e.g., easily readable by the LLM).
    2. They are version-controlled with the code.
    3. They serve as a "Handoff Protocol" between Copilot and Aider.

**Next Level (Future)**:
A truly AI-Native evolution would be a `SessionManager` agent that automatically updates this state based on Git commits and LangGraph execution traces, removing the manual creation step. For now, we manually create the "Memory Anchor".

## Design Decisions
1. **Schema (`domain/schemas/paper.py`)**:
   - Created `Paper` model to standardize how we pass paper data around.
   - Created `OpenAccessInfo` to encapsulate OA-specific details (removing `dict` usage).

2. **Protocol (`domain/ports/fetcher.py`)**:
   - Defined `PaperFetcher` Protocol.
   - Enforced dependency inversion: `infrastructure` will depend on `domain`, not vice versa.

AI 辅助开发框架 2.1：基于 GEMINI.md 的可观测性驱动开发
本指南是您现有 AI 开发规范的全面升级，旨在解决在深度实践中遇到的四大核心挑战：

测试“作弊”：AI 为了通过单元测试而破坏了代码的实际可用性。

上下文迷失：AI 缺乏对项目全局结构的理解，导致反复试错。

手动反馈循环：开发者需要手动复制粘贴大量日志和测试结果。

全局掌控力下降：开发者自身也难以时刻把握项目全貌。

ODAD 框架通过引入“三层测试防护”、“项目知识库 (GEMINI.md)”和“自动化调试上下文”三大核心机制，来系统性地解决这些问题。

第一部分：用“三层测试防护”，杜绝 AI 投机取巧
为了解决 AI “为了通过测试而破坏代码”的问题，我们必须建立一个超越简单单元测试的、多层次的“防护网”。

第 1 层：单元测试 (Unit Tests) - 验证“零件”
目的：确保单个函数或类的内部逻辑正确。这是最小的验证单元。

角色：这依然是 TDD 的基础，但我们必须承认它的局限性——它只关心“零件”本身，不关心“零件”如何被组装。

第 2 层：集成测试 (Integration Tests) - 验证“组件”
目的：确保多个单元（如 Service 与 Repository）能正确地协同工作。这是防止 AI 破坏模块间“契约”的关键。

实践：在您的测试套件中，创建 *.integration.spec.ts 或 tests/integration/ 目录。这类测试应该真实地与数据库、外部 API 的模拟版本 (Mock) 或其他服务进行交互。

对 AI 的约束：集成测试告诉 AI：“我不仅关心你内部的逻辑是否正确，更关心你作为一个‘组件’，对外提供的功能是否符合预期。”

第 3 层：验收测试 (Acceptance Tests) - 验证“产品”
目的：从用户的角度描述一个功能的“业务价值”是否达成。这是最终的、不容置疑的“目标”。

实践：在项目根目录创建 features 文件夹，并使用 Gherkin 语法编写 .feature 文件。这些文件用自然语言描述用户故事。

# features/user_registration.feature
Feature: 用户注册

  Scenario: 一个新用户使用有效的邮箱进行注册
    Given 我在注册页面
    When 我输入一个有效的邮箱 "test@example.com" 和一个强密码
    And 我提交表单
    Then 我应该被重定向到仪表盘页面
    And 一封欢迎邮件应该被发送到 "test@example.com"

对 AI 的约束：验收测试为 AI 提供了最高级别的指令。您的 Prompt 会从“让测试通过”演变为：

Prompt (升级版): “我们的最终目标是让 user_registration.feature 这个验收测试通过。请分析当前失败的集成测试和单元测试，并修改 UserService 的代码来实现这个用户故事。”

附录：正确的开发与测试顺序 (由外向内)
为了最大限度地发挥三层测试防护的威力，推荐遵循以下“由外向内”的开发顺序。

第 1 步：定义“目标” (编写第 3 层：验收测试)
编写 .feature 文件：在开始任何实现之前，首先定义用户的行为和期望结果。

运行并确认失败：运行这个验收测试，它必须失败。这个失败的测试就是本次开发的最终目标。

第 2 步：向内分解，驱动“组件” (编写第 2 层 & 第 1 层：集成/单元测试)
编写失败的集成测试 (第 2 层)：思考需要哪些模块来协作以完成验收测试，并为这些模块间的交互编写一个失败的集成测试。

进入 TDD 循环 (第 1 层)：为了让集成测试通过，开始为单个模块编写单元测试。

红 (Red)：编写一个失败的单元测试。

绿 (Green)：编写最少量的代码让单元测试通过。

重构 (Refactor)：在测试保护下优化代码。

重复 TDD：持续此循环，直到所有必要的单元被开发完成，并让集成测试 (第 2 层) 通过。

第 3 步：验证“目标”，让验收测试通过
再次运行验收测试 (第 3 层)：当所有底层组件都通过了集成测试后，回过头来再次运行最高层的验收测试。

连接并调整：完成最后的“胶水代码”，连接所有组件，直到验收测试最终通过。

这个顺序确保了开发工作始终由业务价值驱动，并为 AI 提供了从宏观到微观的清晰指引。

第二部分：创建 GEMINI.md 项目知识库，赋予 AI 全局视野
为了解决 AI “上下文迷失”的问题，我们采纳官方建议，在项目根目录创建一个名为 GEMINI.md 的文件，作为稳定、持久的“项目长期记忆”。

GEMINI.md 的内容
目的：将项目的所有核心上下文信息整合到一处。这个文件应该包含架构总览、模块地图、核心设计原则等。

示例 (GEMINI.md):

# Gemini 项目上下文指南

## 1. 项目架构总览
- **架构模式**: 本项目采用分层架构 (Controller -> Service -> Repository)。
- **核心目录**:
  - `src/controllers`: 负责处理 HTTP 请求和响应。
  - `src/services`: 包含核心业务逻辑。
  - `src/repositories`: 负责数据持久化。
- **设计原则**: 依赖注入，面向接口编程。

## 2. 模块地图
- **AuthService**:
  - **职责**: 处理用户认证 (JWT 签发与验证)。
  - **依赖**: `UserService` (查询用户)。
- **UserService**:
  - **职责**: 管理用户数据。
  - **被依赖**: `AuthService`, `OrderService`。

如何使用知识库
在您的开场白或关键指令中，主动引用该文件，激活 AI 的“全局记忆”。

Prompt (升级版): “你好 Gemini。在开始之前，请先阅读项目根目录下的 GEMINI.md 文件来理解我们的项目架构和模块设计。现在，我需要在 OrderService 中添加一个新功能，它需要与 UserService 交互。”

第三部分：创建“自动化调试上下文”，告别手动复制粘贴
这是解决您反馈效率问题的核心机制。我们将创建一个脚本，在测试失败时，自动将所有相关信息打包成一个文件，供 AI 直接“消费”。

步骤 1：创建调试脚本
在您的 package.json (或 Makefile/Rakefile 等) 中添加一个脚本。

package.json 示例:

"scripts": {
  "test": "jest",
  "test:debug": "npm test || node scripts/collect_debug_context.js"
}

这个命令会先尝试运行测试。如果测试失败 ( || )，则会执行 collect_debug_context.js 脚本。

步骤 2：编写 collect_debug_context.js 脚本
这个 Node.js 脚本会收集所有调试信息并写入一个文件。请将其放置在 scripts/ 目录下。

全新的调试工作流
在 VS Code 终端中运行 npm run test:debug。

测试失败后，脚本自动运行，生成 .ai-debug/debug_context.log 文件。

您的 Prompt 变得极其简单和高效：

Prompt (终极版): “@workspace 测试失败了。请分析 .ai-debug/debug_context.log 文件，找出根本原因，并直接提供修复后的代码。”

第四部分：引导 AI 成为“领航员”，提升开发者掌控力
最后，我们通过改变提问方式，让 AI 从一个“代码手”，转变为一个能帮助您熟悉项目的“领航员”。

使用“解释-实现-反思”模式
将您的指令分解为三步，强迫 AI 思考并解释。

Prompt (领航员模式):

(解释) “我要在 OrderService 中实现一个取消订单的功能。根据 GEMINI.md 中的信息，请先用文字解释这个功能会涉及到哪些模块的交互，以及你打算如何修改代码。”

(实现) “你的计划听起来不错。现在，请根据你的计划生成具体的代码。”

(反思) “代码已生成。请再次确认，这次修改是否遵循了 GEMINI.md 中定义的设计原则？有没有潜在的副作用？”

这种模式不仅能产出更高质量的代码，更重要的是，它在辅助您开发的同时，也在不断强化您自己对项目全局的理解和掌控。

总结
ODAD 框架是对您现有流程的一次重大升级。通过构建三层测试防护、GEMINI.md 项目知识库、自动化调试上下文，并采用领航员式的交互模式，您可以真正将 Gemini Assistant 深度整合到您的工作流中，解决高级阶段的开发难题，实现前所未有的开发效率和代码质量。

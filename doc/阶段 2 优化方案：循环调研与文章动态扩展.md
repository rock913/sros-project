阶段 2 优化：从“文献循环”转向“稿件循环”

1. 核心逻辑变更：以稿件为状态中心 (State-Centric)

目前的 LangGraph 状态主要存储 candidate_papers。
优化建议：将 manuscript (稿件) 作为状态机的核心。调研的目的是为了“填补稿件中的空白（Gap）”。

循环流程设计：

初始生成：根据用户意图生成“动态大纲（Dynamic Manuscript Outline）”。

空白检测 (Gap Analysis)：AI 扫描大纲，发现某章节缺乏证据支持（如：“需要 3 篇关于 Transformer 内存优化的近三年文献”）。

针对性调研：触发 Discovery Loop（Librarian + Analyst）只针对这个 Gap 寻找资源。

内容扩展 (Incremental Writing)：将调研结果转化为段落，插入稿件，并自动更新引文。

用户引导 (Human-in-the-loop)：用户在 VS Code 中修改段落，AI 观测修改意图，重新进入步骤 2。

2. 关键组件升级 (Phase 2 补丁)

A. 引入 "Gap Finder" 节点 (LangGraph)

职责：不再是盲目搜索，而是比对“当前已有的知识图谱”与“文章写作目标”，列出目前文章最单薄、最需要文献支撑的 3 个方向。

交互：通过“决策卡片”展示给用户：“我发现第二章关于‘计算复杂性’的论证缺乏实验数据，是否现在去检索相关论文？”

B. "Co-STORM" 的实时写作化

文档现状：目前的 STORM 主要是对话。

优化目标：将对话的输出直接映射为 Markdown 的各个 Section。

实现：每一个“Perspective（视角）”节点对应 Markdown 的一个 H2 或 H3 标题。当 Agent 讨论完一个视角，该章节的“草稿（Draft）”即刻更新。

C. 实时引文映射 (Live Citation Mapping)

功能：在阶段 2 就要打通 Zotero。当 Discovery Loop 发现一篇好文章并决定用于“扩展文章”时，立刻在 VS Code 文档底部自动添加引文条目，并插入 [@citekey]。

3. 针对“逐步扩展”的创业思路建议

1. 打造“生长感”的 UI (The Growing Doc)

不要给用户一个空的对话框。界面左边是“生长中的大纲”，中间是“正在撰写的正文”，右边是“Agent 正在检索的动态”。

卖点：让科研人员感觉文章是像植物一样“长”出来的，而不是“憋”出来的。

2. 差异化策略：解决“文献综述”的痛苦

市面上很多工具能写全文，但质量极差。你的机会在于**“局部扩展”**：

用户选中一句话，点击右键“扩充论据”。

Agent 触发：检索 -> 阅读 -> 总结 -> 插入并标注引文。

这种**“小步快跑”的局部循环**比一次性生成全文更有商业价值，因为用户拥有完全的掌控权。

3. 技术壁垒：CiTO 引导的逻辑扩写

利用你文档中提到的 CiTO 本体论。

用户可以命令：“针对这个观点，找一篇**反驳（critiques）**它的文章来增加论证的深度。”

这是目前通用 Agent（如 Cline）完全做不到的逻辑深度。



1. 阶段 2 的合理性评估

目前的阶段 2 规划非常扎实，但存在一个“职能错位”： 在你的文档中，阶段 2 的目标是“Alpha - Discovery Loop”，重点在于“搜索和思维导图”。而你想要达到的“撰写更新”和“文章扩展”被放在了阶段 3 的“Writer's Room”。

如果按照目前的规划执行，阶段 2 结束后，你只能得到一个“增强型的文献搜索器”，无法实现边调研边写的循环。

结论： 阶段 2 需要进行“功能前置”调整，将“初稿生成（Drafting）”与“调研（Discovery）”解耦并建立强连接。

2. 针对目标的架构优化建议

为了实现“循环调研 + 逐步扩展文章”，我建议将阶段 2 升级为 “稿件驱动型调研 (Draft-Driven Discovery)”。

3. 给你的技术落地建议

关于阶段 2 的具体代码实现，你可以参考以下步骤：

State 定义：在 AgentState 中增加一个 draft_sections: Dict[str, str]，Key 是章节 ID，Value 是当前内容。

协同引导：利用 interrupt。每当 Agent 检索完一组文献，先不要写，弹出一张“决策卡片”：“我已经掌握了这 3 篇核心文献，它们分别从 A、B、C 三个角度支持了你的论点，你希望我优先扩写哪个角度？”

动态大纲协议：由于你使用的是 VS Code 插件，利用 FileSystemWatcher 监听用户对 Markdown 大纲的操作。如果用户手动删掉了一个章节，LangGraph 的任务队列应自动剔除该章节对应的调研任务。

总结： 你的规划非常棒，只需要把**“写”的动作从阶段 3 提前到阶段 2 的每一次小循环**中，就能完美实现你“逐步扩展成完整文章”的目标。
